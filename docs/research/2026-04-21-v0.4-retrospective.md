## New risks discovered during execution (not in §8.1)

### 1. Orphan release-asset upload (release.yml:72-77)

**What:** `release.yml` uploaded `installer/install-cli.sh` as a release asset under the assumption of a `curl | bash <release-url>` install flow. Docs in `installer/install-cli.md` pivoted to `git clone` in commit `267ba27` citing the `BASH_SOURCE[0]` problem, but the release-asset upload was not removed at the same time.

**How found:** Side-session review mid-execution, before tagging.

**Fix:** Commit `d9f2687` removed the upload block + "Verify installer asset exists" pre-check + comment explaining why. Asset was an unadvertised footgun (anyone who discovered the URL would get the `BASH_SOURCE[0]` failure).

**Lesson:** When a locked design decision flips mid-execution, grep every consumer of that decision's artifacts. The plan's "Locked: Public install URL = GitHub release asset pinned at tag" should have surfaced this as a cross-reference check during commit `267ba27`.

### 2. Release workflow permission regression (same commit)

**What:** The release-asset removal commit (`d9f2687`) dropped `contents: write` from the workflow's `permissions:` block. GitHub Actions' default behavior when ANY permission is specified: unspecified permissions default to `none`. So removing `contents: write` without adding `contents: read` dropped both — `actions/checkout` then 404'd because it had no read access.

**How found:** Tag `v0.4.0-alpha` pushed; release workflow failed at checkout within 30 seconds; verified via `gh run view --log-failed`.

**Fix:** Commit `bb7b65e` added `contents: read` back. Tag moved from the first merge commit (`b59fb19`, broken workflow) to `bb7b65e` (fixed workflow). GHCR publish fired correctly on re-tag.

**Lesson:** GitHub Actions' `permissions:` block is not subtractive — specifying one permission silently narrows all others to `none`. Safer pattern: `permissions: read-all` + selective overrides, or explicitly list every permission the job actually needs. Also: test workflow changes via a non-release tag first when feasible (e.g., `v0.4.0-alpha-rc1`).

### 3. `gh pr merge --delete-branch` worktree collision

**What:** `gh pr merge --squash --delete-branch` tries to check out `main` locally after the GitHub-side merge completes. Failed because `main` was checked out in another worktree (`E:\Projects\universal-memory`). The GitHub-side merge had already succeeded; the local post-merge step errored with `fatal: 'main' is already used by worktree`.

**Impact:** Cosmetic. Remote branch was cleaned up, merge commit landed on main correctly. Just had to `cd` to the main worktree and pull manually.

**Lesson:** For worktree workflows, don't rely on `gh pr merge --delete-branch` to do local cleanup. Either do the merge from the main worktree, or accept the harmless local-checkout error.

## Lessons for planning the next release

1. **Locked decisions need a cross-reference audit when they flip.** The `curl | bash install-cli.sh` URL was "locked" in the plan and in the handoff. When `267ba27` flipped the docs, release.yml's assumption didn't get re-examined. A grep for the old decision's artifacts (the release-asset URL string, the upload step, the `contents: write` permission) would have surfaced the orphan immediately. Next release: when a locked decision flips, search for every artifact that embodied the decision.

2. **GitHub Actions `permissions:` is subtractive in disguise.** Specifying any permission silently narrows all unspecified permissions to `none`. When editing a workflow's `permissions:` block for one concern, also enumerate every permission the job's remaining steps need. Or use `permissions: read-all` + selective writes.

3. **Three Opus review rounds was the right cost for this scope.** Round 1 = 18 findings; round 2 = 11; round 3 = 23 (driven by Phase D's fresh shell surface). Convergence held. A fourth round was considered and rejected by the convergence trendline — reviewer fatigue would have lowered signal-to-noise. Final code-review pass pre-merge caught 3 Important + 4 Nit findings that round 3 missed (because round 3 was pre-Phase-E).

4. **Dual-Opus review pattern validated.** Two reviewers with different lenses (correctness vs. scalability) caught disjoint findings in rounds 1–2. Single reviewer would have missed ~30% of design-gap findings based on the per-round gap-overlap measurement.

5. **Subagent-driven development + independent code-reviewer pre-merge** is the right quality gate for doc-heavy phases. Phase E's 11 commits landed at writer→reviewer quality without me re-writing them manually. The final review caught what subagent self-review missed.

6. **CC-augmented timeline dramatically collapsed wall-clock.** Plan estimated 3–5 weeks median; actual shipping took a handful of sessions. Framework for v0.5 planning: estimate in session-count, not days.

## Post-ship findings (after tag, before next release)

### Marker-block idempotency — unbounded bashrc growth on re-install

**What:** `installer/lib/marker-block.sh:62` prepended a leading `\n` on every write, but the awk-strip step kept the blank line that the PREVIOUS run had written. Re-running `install.sh` or `install-cli.sh` grew `~/.bashrc` by 1 blank line per invocation.

**How found:** Post-release VM smoke test (Ubuntu 24.04 Docker container, bind-mounted `v0.4.0-alpha` tarball, pre-seeded `~/.bashrc` + `~/.local/bin` with non-trivial user content). CI's installer matrix starts from an empty HOME where the bug is invisible on the first run — only re-install re-run exercises the accumulation path.

**Fix:** `awk` now buffers blank lines and discards the buffer when it sees the marker-start sentinel. Preceding blank lines drop; the new block adds exactly one separator. Verified stable across 4 consecutive re-runs.

**Status:** Fixed on `main` (commit `46e8700`); NOT back-ported to the `v0.4.0-alpha` tag. Users who re-install from the tag hit the bug; users cloning `main` get the fix. Alpha-acceptable delta.

**Regression coverage added:**
- `installer/install-cli.test.sh` — new T8 (3 re-runs → exactly 1 marker block + line-count stability between runs 3 and 4).
- `server/install.test.sh` — T18 extended with a 4th run + line-count stability assertion.

**Lesson for future v0.x installer tests:** the idempotency check must assert line-count stability, not just marker-count uniqueness. v0.4 CI's `marker-start == 1` and `marker-end == 1` assertions passed under the bug because only the *count* of sentinels was checked, not the accumulation of blank lines between them.
